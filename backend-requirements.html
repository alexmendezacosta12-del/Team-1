Core Functional Requirements

Core
CFR-01 Display product items from a live database Partial
Products are currently held as a static JavaScript array. This must be replaced with a live database connection so that product additions, removals, and stock updates are reflected on the site in real time.
REST API endpoint GET /api/products returns paginated product list with filters as query params
Database schema: products table with fields: id, name, category, colorway, price, stock_qty, sizes (JSON), image_url, featured, release_date, collection (sport | formal)
Both Sport and Formal collections served from the same database, distinguished by collection field
Admin interface for adding/editing/removing products and updating stock levels
CDN or cloud storage (e.g. AWS S3) for product image assets
CFR-02 Product search functionality Implemented
A search bar is present on both the Sport Shop and Formal Shop pages. Currently searches client-side across the static product array. When CFR-01 is live, search must be backed by server-side full-text search.
API endpoint GET /api/products/search?q={query} — searches name, colorway, category, description
Debounced input (300ms) to avoid excessive requests
Search results scoped to the active collection (sport or formal)
Graceful empty state when no results are found
2

Stretch Requirements

Stretch
STR-01 Filters — size, colour, price, category Implemented
Client-side filtering is fully implemented on both shop pages (colour, size, price range, style category). When connected to a live database, filters should be passed as server-side query parameters to avoid loading the full catalogue client-side.
Filter params passed to GET /api/products?color=&size=&minPrice=&maxPrice=&category=
Filter state preserved in URL query string for shareable links
Filter options (available colours, sizes) driven from live inventory — out-of-stock sizes shown as disabled
STR-02 Customer login and account functionality Pending
No user authentication is currently implemented. A full account system is required for the 10% sign-in discount banner shown on the sport landing page and for order history access.
Endpoints: POST /api/auth/register, POST /api/auth/login, POST /api/auth/logout, GET /api/auth/me
JWT or session-based authentication with secure HTTP-only cookies
Account dashboard: order history, saved addresses, wishlist
Sign-in discount (10%) applied automatically at checkout via coupon code linked to verified account
Password reset flow via email (transactional email provider e.g. SendGrid)
OAuth sign-in option (Google) as stretch-of-stretch
STR-03 Checkout journey — basket and payment Partial
The basket (add to cart, quantity controls, remove, clear) and checkout form UI are fully implemented on both Sport and Formal shop pages. Cart state persists via localStorage. Payment processing and order persistence are not yet connected.
Cart state migrated from localStorage to server-side session or user account once STR-02 is live
Endpoint POST /api/orders — creates order record, triggers stock decrement, returns order ID and confirmation
Payment integration via Stripe (card) — use Stripe Elements on the checkout form; no raw card data handled server-side
Order confirmation email sent on successful payment (transactional email)
Endpoint GET /api/orders/:id for order status lookup
Webhook handler POST /api/webhooks/stripe for async payment events (refunds, failures)
Stock validation at checkout — prevent overselling if stock changes between add-to-cart and payment

Future Development

Future
FUT-01 Personalised recommendations Partial
A rule-based AI Shoe Finder is already implemented on the Sport shop page — it scores products against user-provided height, weight, colour preference, and foot type. Future development should replace this with a machine-learning recommendation engine.
Collaborative filtering model trained on order history — "customers who bought X also bought Y"
Endpoint GET /api/recommendations?userId={id} returns ranked product list
Cold-start fallback (no history): use the existing rule-based scorer
A/B testing framework to compare recommendation strategies
Extend to Formal collection with style-affinity signals (e.g. Oxford preference score)
FUT-02 AI-powered assistance Pending
A conversational AI assistant embedded on the site to help customers find the right shoe, answer questions about materials and sizing, and support post-purchase queries.
LLM-backed chat widget (e.g. OpenAI GPT-4o with function calling) — surfaced as a floating button on all pages
System prompt grounded on live product catalogue (RAG — retrieval-augmented generation)
Functions available to the model: search_products, get_order_status, get_size_recommendation
Conversation history stored per session; linked to user account if authenticated
Escalation path to human support (Contact page) when the model signals low confidence
Rate limiting and cost controls on the LLM API to prevent abuse

Proposed Tech Stack

Layer Technology Rationale
Frontend HTML / CSS / Vanilla JS (current) No framework dependency; lightweight and fast. Upgrade to React or Next.js if component complexity grows.
Backend API Node.js + Express (or Next.js API routes) Matches JS skillset already in use on the frontend. Fast to prototype REST endpoints.
Database PostgreSQL Relational model suits products, orders, users, and inventory. Strong JSON support for sizes array. Hosted on Railway or Supabase for ease.
Authentication NextAuth.js / Passport.js Handles session management, JWT, and OAuth providers out of the box.
Payments Stripe PCI-compliant, well-documented, supports UK card schemes and strong customer authentication (SCA).
Image Storage AWS S3 + CloudFront CDN Scalable image hosting with edge caching. Replace inline SVG placeholders with S3-hosted photography.
Email SendGrid (transactional) Reliable delivery for order confirmations and password resets. Free tier sufficient at launch volume.
Deployment Vercel (frontend) + Railway (API + DB) Zero-config deployments from Git. Separate frontend and API concerns for scalability.
AI / Recommendations OpenAI API + pgvector GPT-4o for the chat assistant; pgvector extension on PostgreSQL for embedding-based product search (RAG).
